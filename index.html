FriendsConnect

This project was developed by David Levitsky at California Polytechnic State University, San Luis Obispo, as part of CPE 305 - Individual Software Design and Development.

Background

Connecting with others has become a big focus in today's fast-paced day and age. While there are many different applications to help facilitate interactions between two different users, they all operate separately and on their own premise. This application strives to integrate pre-existing services using developer API's to allow users to schedule events with their friends in a simpler and quicker manner than the status quo. After all, when our smartphones have more capabilities than the computers that landed us on the Moon, shouldn't inviting a friend out be a breeze?

Some key features:

  • Calendar view
  • Yelp Search Integration
  • E-Mail Notification
  • Dynamic Upcoming Schedule Viewer
  • History of Past Events
  • Feedback Mechanism for Previous Events
  • Persistence of Data through Realm Database

Basic Walkthrough

Main Screen - the application will begin at this homepage.

Creating an Event

To create an event, click the green "plus" on the Main Screen. Fill out the appropriate fields.

Using Yelp

If you're not sure about a good place to venture with a friend, whether it be for a dinner, museum, or a run, FriendsConnect provides simple access to Yelp without forcing you to open up a separate application. Simply type in a search term, like "dinner", and your preferred location, such as "San Francisco", and choose from generated results!

Viewing Additional Details

You can click on the information button under the image of each result to view additional details, such as address and number of user reviews.

Confirming Your Event

To select a location, click on the checkmark under the image of the Yelp result you wish to select. Upon returning to the page you were originally at to create an event, edit any other details you wish and click "SAVE".

E-Mail Notification

An automatically generated e-mail will be composed for you, and you will be directed to the e-mail service you use on your phone. Edit any details you wish, and click send to notify your invitee! Their e-mail address will already be typed in for you.

That's it!

You will receive a brief popup confirming the scheduling of your event, and it should now appear on your schedule of Upcoming Events. Click on a different day on the Calendar to have it adjust the daily schedule of Upcoming Events based on the selected date.

Rate Your Previous Experiences

After attending an event, you can click the star on the home page to be taken to a list of past events. Use the feedback feature of thumbs up / thumbs down so you can remember your experience for the future!

Technical Details

High-Level Architecture

Android operates on a Model-View-Presenter architecture. This architecture is highly effective because screens could change at any time, based on a button press or some user input.

  • View is a layer that displays data and reacts to user actions. On Android, this could be an Activity, a Fragment, an android.view.View or a Dialog.
  • Model is a data access layer such as database API or remote server API.
  • Presenter is a layer that provides View with data from Model. Presenter also handles background tasks.

You can see a high-level overview below. Note that the Presenter is the only method of communication between the View and the Model, and that the two should never be able to communicate directly.

Design Architecture

Here you can see the design structure of the project, complete with class diagrams, activities, and XML files. The MVP structure was slightly modified to add a persistence layer, which is a Realm database used to store created events.

The Model layer describes the core data structures and logic. The EventsList is the main container of information, since it holds all past and future Events that a user has scheduled. ContactsList is a Singleton representation of the user's contacts pulled from their phone after permission is granted. YelpResults are used to store information created by the YelpAPI.

The Presenter layer outlines the different screens that the user will see, specifically the various Activities. Each Activity serves its own purpose. By using Intents, the Activities are able to communicate with each other and create data flow.

The View layer lists the XML files used for visual representation of the aforementioned Activities.

Design Patterns

The Singleton pattern was used twice in this project - one to create an instance of EventsList, which is a tangible Java representation of information stored in a Realm database, while the other is an instance of ContactsList, which is responsible for storing a user's contacts. The benefits of using a Singleton pattern was to improve the organization and performance of the application. Re-loading the user's contacts consistently would become an expensive operation, so doing it once and storing the result in a Singleton class prevents redundancy. For the EventsList class, allowing multiple instances of a database representation could cause serious bugs if different threads were operating on the database simultaneously.

Developer's Notes: Lessons Learned for the Future

Architecture

Although the initial architecture diagrams developed prior to writing the application were decently structured, I ended up needing a lot more classes than I originally planned. While writing the code, I focused too much on implementation and did not revisit the architectural design as much as I should have. Therefore, although the MVP structure still works fine, the class diagram in the model section could have been much cleaner and more modular. Design Patterns

Using the Singleton design pattern helped greatly to organize pieces of data that remained fairly constant. However, my communication between the Realm Database and the EventsList Singleton could have been improved by using an Observer pattern. As proposed in the class diagram above, I would have added an EventListener to modify the Realm Database whenever an Event was added/deleted from the EventsList instance.

Future Features

The application could benefit greatly from integrating the Google Maps API so a user can immediately look up directions to an event directly from the application. The usage of the Yelp API could be flushed out to take into consideration additional factors such as price. Adding an option to allow notes to be attached to an invitation would be helpful. As always, the User Interface could be more appealing to the eye.

Concluding Thoughts

Overall, I enjoyed the process of designing a system, implementing it, and then analyzing the final product. I was able to use several design patterns to establish a sustainable and structurally sound architecture, which benefitted the readability and performance of the application. I really enjoyed working with the Yelp API, as it was very easy to use and provided great data. I learned from my mistake of focusing only on implementation rather than revisiting the architectural design once I needed additional data models, and I look forward to improving my software design skills with future projects.
